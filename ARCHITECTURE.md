# System Architecture & Technical Design Document

## 1. Executive Summary
The **Yuno AI Payment Orchestrator** is an enterprise-grade, event-driven multi-agent system designed to resolve failed payments and combat involuntary churn. Rather than relying on rigid, hardcoded email sequences or a single monolithic LLM prompt, this system implements a deterministic **LangGraph** state machine. This architecture orchestrates five specialized AI micro-agents, ensuring that every customer interaction is highly personalized, context-aware, and strictly compliant with business policies.

---

## 2. The Business Problem: Involuntary Churn
In modern SaaS and e-commerce platforms, up to 40% of churn is involuntary—caused by expired cards, strict fraud filters, or temporary gateway timeouts. Traditional recovery methods (e.g., generic "Payment Failed" emails) yield low conversion rates because they lack context and empathy. 

This system solves that by treating payment recovery as a dynamic, intelligent routing problem. It analyzes the *reason* for the failure and crafts a localized, empathetic response tailored to the specific user and error code.

---

## 3. Core Architectural Principles
- **Deterministic State Management:** The system passes a strict, immutable `State` dictionary between nodes. This ensures that no data is lost or hallucinated during the multi-agent handoffs.
- **Separation of Concerns:** Instead of one massive LLM handling logic, routing, and drafting, the system uses specialized micro-agents. A "Planner" agent only routes, a "Customer" agent only drafts, and a "Compliance" agent only audits.
- **Zero Hallucination Leakage:** Generative AI is inherently unpredictable. To safely deploy this in a financial context, an independent Compliance Agent acts as a firewall, evaluating all outgoing drafts before they reach the customer.

---

## 4. Multi-Agent Workflow (The LangGraph Nodes)

The orchestration graph executes sequentially, passing the state payload through the following dedicated nodes:

### A. Event Ingestor (`ingest_event`)
- **Role:** The entry point for the webhook payload.
- **Function:** Parses the raw incoming data, normalizes it, and extracts the core parameters (User ID, Amount, Error Code, Gateway Response). It immediately strips out highly sensitive PII (like full credit card numbers) before the data ever touches the LLM context window.

### B. Reason Classifier (`classify_reason`)
- **Role:** The analytical engine.
- **Function:** Ingests raw banking error codes (e.g., `51: INS_FUNDS` or `05: DO_NOT_HONOR`) and translates them into actionable human business logic. It determines whether the issue is a hard decline (requires a new card) or a soft decline (can be retried later).

### C. Channel Planner (`plan_strategy`)
- **Role:** The tactical decision-maker.
- **Function:** Evaluates the user's metadata (geography, LTV, tier) and the failure severity to determine the optimal recovery channel. For instance, high-value urgent failures are routed to Telegram/SMS, while standard timeouts are routed to Email.

### D. Customer Agent (`draft_message`)
- **Role:** The empathetic communicator.
- **Function:** Receives the classified reason and the planned strategy, then drafts a concise, highly personalized message. It is strictly instructed to avoid technical jargon, prioritize empathy, and provide a single, clear call-to-action (e.g., a secure link to update payment methods).

### E. Compliance Agent (`check_compliance`)
- **Role:** The security firewall.
- **Function:** Independently audits the draft generated by the Customer Agent. It checks for:
  - Exposure of raw error codes or internal IDs.
  - Aggressive or inappropriate tone.
  - Missing calls-to-action.
- **Routing Logic:** If the draft passes, the graph routes to the final dispatch node (`END`). If it fails, the graph dynamically loops back to the Customer Agent with a `REVISION_REQUIRED` flag, forcing a re-draft.

## 5. Technology Stack Rationale

| Technology | Purpose & Justification |
| :--- | :--- |
| **LangGraph** | Chosen over standard LangChain chains because it allows for cyclical graphs (essential for the Compliance revision loop) and strict, explicit state management between nodes. |
| **FastAPI** | Provides asynchronous, high-throughput webhook handling, enabling the backend to ingest thousands of events simultaneously without blocking the event loop. |
| **React + WebSockets** | Delivers a real-time Orchestration Dashboard. WebSockets are utilized to stream the LangGraph execution logs live to the frontend, providing critical observability for operations teams. |
| **Groq (Llama 3.3 70B)** | Utilized for LLM inference due to its ultra-low latency architecture via LPUs (Language Processing Units). This allows the entire 5-agent graph to execute in seconds, enabling near real-time customer outreach. |
| **Tenacity** | Implements exponential backoff and retry logic across all external API calls (LLM and Telegram), ensuring high fault tolerance against network jitter. |

## 6. Conclusion
The Yuno Orchestrator demonstrates a paradigm shift in how AI can be deployed in financial operations. By wrapping generative LLMs in deterministic, safety-critical workflows, this architecture proves that AI can safely and autonomously handle sensitive customer recovery operations at an enterprise scale.
